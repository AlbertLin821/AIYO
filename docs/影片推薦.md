# 那這樣的話我要問幾個問題，影片是由AI決定的?youtube推薦的? 要怎麼樣才可以確保"真正找到符合使用者需求的影片推薦給使用者"?要怎麼正確的把擷取出來的景點讓AI或是用功能寫入到網站中的行程規劃畫面讓使用者看到，整體的體驗我想要是順暢的行程規劃 使用者只須要用講話 看影片 看地圖中的景點 就可以規劃出行程內容。

先把結論講在前面：

- 影片不是「完全交給 YouTube 決定」，而是 **YouTube Data API 先提供候選清單，你自己的系統再用規則＋embedding＋AI 做第二層排序**，最後推薦給使用者。[^1][^2][^3]
- 「真正符合需求」的關鍵是：**硬條件過濾（目的地、語言、長度、品質）＋語意相似度（偏好向量 vs 影片片段）＋使用者回饋**，而不是單一演算法。[^4][^5][^2][^6]
- 把擷取出的景點寫進行程畫面，做法是：**批次管線先把「影片片段→景點→座標」寫進資料庫，再由行程規劃 API 把這些 place_id / segment_id 填入行程 JSON，前端只要照 JSON 畫地圖＋影片就好**。[^7][^1]

下面用你問的三個問題逐一拆開。

***

## 影片是 AI 決定的，還是 YouTube 決定的？

實務上會是「兩階段」：

1. **YouTube 當「搜尋引擎」，先給一堆候選影片**
    - 用 YouTube Data API 的 `search.list` 搭配 `q=目的地關鍵字`（例：`\"Tokyo travel vlog\"`），你可以決定排序方式，例如 `order=relevance` 或 `order=viewCount`。[^8][^1]
    - 同時可以加上 `regionCode`（如 `JP`）與 `relevanceLanguage=zh-Hant`，限制影片國別與語言相關性。[^1]
    - 接著用 `videos.list` 拉出每支影片的 `snippet`、`statistics`（觀看數、評分）、`contentDetails`（長度）等詳細資訊。[^9]
→ 這一層只是「抓候選」，還不算真正的推薦邏輯。
2. **你的系統＋AI 在候選上做「第二層決策」**
在候選集合上，你用以下條件做篩選＋排序：
    - 硬條件過濾：
        - 必須在「指定城市/景點」附近（由字幕 NER 出來的地點＋地理編碼比對）。
        - 影片長度在合理範圍，例如 5–30 分鐘。
        - 觀看數與 like ratio 達到一定品質門檻（例如 viewCount > X，likeCount/viewCount > 某比例）。[^10][^9][^1]
    - 語意相關性：
        - 將每個影片片段的字幕轉 embedding（你已經有 nomic-embed-text + pgvector），也為「使用者偏好＋這次的語音需求」算一個 embedding，做 cosine similarity 排序。[^5][^2][^6]
    - 多樣性控制：
        - 即使相關度高，也不要都推薦同一類型（例如全都是美食），可以在排序時加一個「類型多樣性」的規則，避免看起來很單調（這部份可以先用 heuristic，不一定要 model）。

所以可以對教授說：
> 「YouTube 只負責提供候選影片，我們再根據使用者即時需求、長期偏好、影片字幕內容和品質指標，做第二層排序，確保推薦結果是『對這個人』有意義的，而不是單純 YouTube 全站熱門。」

***

## 怎麼做才能「真的找到符合使用者需求的影片」？

可以想成一個「多訊號推薦模型」，你不一定一開始就寫 ML model，先用「加權分數」就很夠：

### 1. 先確保「主題對、地點對」：硬條件過濾

這一層只處理「這支影片到底是不是講這個地方」：

- 在批次管線裡，你已經：

1. 從影片字幕 / Whisper 轉寫得到段落文字。
2. 用 Ollama NER 模型抽出地點實體（城市、景點）。
3. 用 Geocoding API 把地點轉 lat/lng，寫入 `segment_places` 表。[^11][^7]
- 查詢時（例如使用者說「我想去東京」）：
    - 只選擇那些其 `segment_places` 裡 city/country 是東京 / 日本，或座標落在東京 bounding box 內的影片與片段。
    - 這樣可以排除「只是標題寫 Tokyo，但內容在講其他地方」的情況。


### 2. 再用「語意相似度」對齊使用者需求

這一層處理「這支影片是不是這個人喜歡的風格」：

- 建立使用者偏好向量：
    - 依你之前做的記憶系統，將「他常問的詞、收藏的景點、看過的影片標籤（美食、夜景、親子）」轉成 embedding，平均或加權平均成一個 user embedding。[^2][^6]
- 每個 `video_segment` 也有其 embedding。
- 分數計算可以類似：

$$
score = 0.5 \cdot \text{location\_match} + 0.3 \cdot \text{semantic\_similarity} + 0.2 \cdot \text{quality\_score}
$$
    - `location_match`: 影片與目標城市的契合程度（完全在東京=1，附近城市=0.5，其他=0）。
    - `semantic_similarity`: user embedding 與片段 embedding 的 cosine similarity。
    - `quality_score`: 正規化後的 viewCount、like ratio、發布時間等綜合指標。[^6][^4][^5][^2]

然後取 top N 影片，再從中挑選每支影片的幾個代表性片段。

### 3. 加上「使用者回饋」迴圈

前面都是「離線訊號」，進一步還可以用互動行為調整：

- 記錄：使用者有無點開、觀看超過某比率、是否收藏、是否標記「不喜歡」等等。
- 每次產生推薦時，可以略微提高「他之前看過且停留時間長」之類影片類型的權重，降低他曾經關掉很快的類型，慢慢個人化。[^2]

即使一開始沒有複雜模型，單靠這種「規則＋相似度＋回饋」的設計，教授就會看到你真正有在思考推薦系統，而不是只丟給 YouTube。

## 把景點寫進行程畫面的正確做法（資料流＋UI）

關鍵是：**行程畫面顯示的是「資料庫裡已經被整理好的結構化資料」**，不是讓 LLM 現場亂講完就算。你可以這樣設計：

### 1. 資料層：幾個核心 table／關聯

舉例（概念即可，實作時依你實際 schema 微調）：

- `places`
    - `id`（place_id, PK）
    - `name`（淺草寺）、`city`、`country`
    - `lat`、`lng`
    - `category`（temple, shopping, food…）
- `videos`、`video_segments`（你已經有雛形）
    - `video_segments`：`id`（segment_id）、`video_id`、`start_sec`、`end_sec`、`text_merged`
- `segment_places`（多對多關係）
    - `segment_id`、`place_id`（連到 `places.id`）、`importance` 等。
- `itineraries`
    - `id`（itinerary_id）、`user_id`、`destination`、`days`、`created_at`…
- `itinerary_stops`
    - `id`
    - `itinerary_id`
    - `day_index`（第幾天）
    - `order_index`（當天第幾個點）
    - `place_id`（連到 `places.id`）
    - `video_segment_id`（可選，連到某個影片片段）

這樣一來，「景點」跟「影片片段」是被 normalize 串起來的，前端只要根據這些 id 去畫 UI。

### 2. 行程規劃時，AI 不直接寫 HTML，而是「填這些 id」

當使用者語音說完需求後，你的 `plan_trip_voice` API 可以這樣工作：

1. 依目的地，從 `places` 抓出候選景點（已經有大量影片片段支援的那一些）。
2. 用交通距離、類型分佈、使用者偏好等規則，先在程式裡做一個「初步路線」（例如每天 4–6 個點）。
3. 把這個候選列表（含 place_id、緯經度、關聯的 video_segment_id）丟給 LLM，讓它做：
    - 微調順序（在可接受範圍內）。
    - 幫每一天寫 summary 文案和說明，輸出格式仍然是 JSON：

```json
{
  "days": [
    {
      "day": 1,
      "summary": "...",
      "stops": [
        { "placeId": 10, "videoSegmentId": 123 },
        { "placeId": 15, "videoSegmentId": 456 }
      ]
    }
  ]
}
```

4. 後端驗證 JSON schema，然後把 `itinerary` 和 `itinerary_stops` 寫進資料庫。

**重點：LLM 不直接創造地點名稱與座標，而是只在已知的 place_id／segment_id 上做選擇與排序**，這樣行程畫面才能穩定、可重現，也不會出現地圖上找不到的虛構景點。[^7][^2]

### 3. 前端行程畫面的渲染

目前你想要的體驗是：「只用講話、看影片、看地圖，就把行程排出來」，那前端可以這樣設計：

1. 使用者講話 → Web Speech API 轉文字 → 呼叫 `/api/plan_trip_voice`。
2. API 回傳的 payload 可以長這樣（簡化示意）：

```json
{
  "itineraryId": 42,
  "days": [
    {
      "day": 1,
      "summary": "第一天在淺草和晴空塔附近散步",
      "stops": [
        {
          "place": {
            "id": 10,
            "name": "淺草寺",
            "lat": 35.7148,
            "lng": 139.7967
          },
          "videoSegment": {
            "videoId": "abc123",
            "startSec": 120,
            "summary": "介紹淺草寺與雷門"
          }
        }
      ]
    }
  ],
  "videos": [
    {
      "youtubeId": "abc123",
      "title": "東京三日遊",
      "channelTitle": "XXX Travel",
      "viewCount": 123456
    }
  ]
}
```

3. 前端據此同時做三件事：
    - **地圖**：把 `days[].stops[].place` 當 marker；按某 stop 就把地圖移過去，並在影片區選對應 segment。
    - **影片區**：渲染 3–5 個影片卡，每個影片有時間軸上可點擊的 `videoSegment` 標記。
    - **行程列表**：左邊或下方顯示每一天的 stops 排序，可以拖拉調整，調整完再 PATCH 回 `/api/itineraries/{id}`。

整個流程對使用者來說就會是：

> 按住講話 → 等 1–2 秒 → 地圖自動飛到目的地，左邊多出一個初步行程，每個景點旁邊都有對應影片片段可以點來看。

背後其實是你已經事先把「影片 → 片段 → 景點 → 座標」全部準備好，線上只是在這些資料上做智慧組合。
